Qatoto Backend · Lesson 01

Your first Express server

Before any login, before any database — a server that answers one request. This is the skeleton every auth route in your backend plan will hang off.

The win: by the end you'll have an Express server running, and a real URL — http://localhost:4000/health — that returns JSON in your browser. That's the foundation. Everything else is more routes on this same shape.

Why this first

Your mission is real auth for Qatoto. But auth is just special routes on a web server: POST /auth/login, GET /auth/me. You can't write those until you can write any route. So today's job is the smallest possible route — GET /health — so the moving parts (server, route, JSON, port) become familiar before passwords and cookies pile on. Small working thing first, always.

Do it

In a terminal, in a fresh folder — not inside the frontend repo.

  1. Make the project

    mkdir qatoto-backend && cd qatoto-backend
npm init -y

    npm init creates package.json — the file that lists your project's dependencies.

  2. Install Express + TypeScript

    npm install express
npm install -D typescript tsx @types/node @types/express

    tsx runs TypeScript files directly — no build step, no config needed to start. The @types/* packages teach your editor what Express looks like, so you get autocomplete and red squiggles on mistakes.

  3. Write the server

    Create a file src/index.ts with exactly this:

    import express, { type Request, type Response } from "express";

const app = express();              // create the application
app.use(express.json());            // teach it to read JSON request bodies

// A "health check" route: GET /health  ->  { data: { ok: true } }
app.get("/health", (request: Request, response: Response) => {
  response.json({ data: { ok: true } });
});

const PORT = 4000;
app.listen(PORT, () => {
  console.log(`Qatoto backend running on http://localhost:${PORT}`);
});

  4. Run it

    npx tsx src/index.ts

    You should see Qatoto backend running on http://localhost:4000. Leave it running. To stop it later: Ctrl + C. (Next lesson we add a tsconfig.json and an npm run dev shortcut that auto-restarts on save.)

See the win

Open http://localhost:4000/health in your browser. You should see: 

{ "data": { "ok": true } }

Or from another terminal: curl http://localhost:4000/health. That JSON came from your server. You built an API.

Read every line you wrote. Can you say, out loud, what app.get(...), request, response, and app.listen(...) each do? If any is fuzzy, that's the perfect thing to ask your teacher before the next lesson — it'll come back in every route.

Words you just met

Server
A program that waits for requests and sends back responses. app.listen starts it waiting.
Route
A rule: "when a method + path like GET /health arrives, run this function." You wrote one.
Endpoint
The address a route lives at — here, http://localhost:4000/health.
Middleware
A function that runs on requests before your route — express.json() is one; later, requireAuth will be another.
Port
The numbered "door" on your machine the server listens at — 4000. The frontend runs on a different one (3000).
JSON
The text format APIs speak in — { "key": "value" }. Your route returned some.
